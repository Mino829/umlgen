# デザインパートナー検証プレイブック

この文書は、umlgenの4週間デザインパートナープログラムを3チームで同じ条件にそろえ、作り手の期待ではなく利用結果で次の機能を決めるための手順です。

## 成功条件

- 3チームへ導入する
- 2チーム以上が4週間、週1回以上利用する
- 2チーム以上が「なくなると困る」または同等の評価をする
- 共通する課題を3件以上抽出する
- 少なくとも1チームへ有料導入または継続サポートを提案する

## 選定

次をすべて満たす応募を優先します。

1. Javaを使用している
2. GitHub Pull Requestでレビューしている
3. 週1回以上Pull Requestを作る
4. GitHub Actionsを利用できる
5. 1週目と4週目にフィードバックできる

同じ会社・構成だけに偏らないよう、可能であれば個人、業務チーム、マルチモジュールの組み合わせを選びます。

各チームで次を記録します。

## 導入前インタビュー

1. 現在、設計変更や依存関係の変化をどう確認しているか
2. 直近でレビュー時に見落とした、または確認に時間がかかった変更は何か
3. 1週間のPull Request数と平均的な変更規模
4. 図を確認したいのは誰か
5. ソースコードを扱うツールに必要なセキュリティ条件は何か

機能説明を先に行いすぎず、現在の行動と実例を確認します。

## 1週目

1. 導入に何分かかったか
2. 生成に失敗したPull Requestはあったか
3. 図を実際に開いたか
4. 図から気付いたことがあったか
5. 次の1週間も自発的に使うと思うか

設定で解決できる問題は調整します。新機能の要望は実装せず、候補として記録します。

## 4週目

1. 図を確認したPull Requestを具体的に思い出せるか
2. 図によってレビューの判断や会話が変わったか
3. なくなった場合に困るか。困らない場合はなぜか
4. `private repo対応`、`PRへの直接コメント`、`履歴`の優先順位
5. 月500円または月1,500円で使うために必要な条件
6. 3,000円の導入・設定サポートを別リポジトリで利用するか

価格の質問は、好意的な反応ではなく具体的な購入条件を確認します。

## 次の機能を作る判断

- 2チーム以上が同じ問題を実際のPull Requestで経験した場合に優先候補とする
- 要望ではなく、発生頻度と回避に使っている時間を記録する
- 2チーム以上がPRコメントを必須とした場合、GitHub App MVPを次の検証対象にする
- 継続利用が1チーム以下の場合、Cloud開発を開始せず、対象ユーザーまたは価値提案を見直す
